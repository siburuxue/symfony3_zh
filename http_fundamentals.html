Symfony和HTTP基础工具¶

恭喜！通过学习Symfony，你就可以成为一个更高效，全面和流行的Web开发人员（实际上，你自己是最后一部分）。 Symfony的建立是为了回归基础知识：开发工具，让您可以更快地开发并构建更强大的应用程序，同时保持您的方式。 Symfony建立在来自许多技术的最佳想法的基础上：您即将学习的工具和概念代表了成千上万的人多年的努力。 换句话说，您不仅仅学习“Symfony”，您正在学习Web的基础知识，开发最佳实践，以及如何在Symfony内部或独立于Symfony使用许多令人惊叹的新PHP库。 所以准备好了。

真正的symfony的理念，本章首先解释的基本概念，Web开发常见：HTTP。无论您的背景或首选编程语言，本章都是必读的。

HTTP很简单¶

HTTP（极客的超文本传输协议）是允许两台机器相互通信的文本语言。 是这么回事！ 例如，当检查最新的xkcd漫画时，发生以下（近似）对话：

虽然使用的实际语言更为正式，但它仍然非常简单。 HTTP是用来描述这种简单的基于文本的语言的术语。无论您在网络上如何开发，您的服务器的目标始终是了解简单的文本请求，并返回简单的文本响应。

Symfony是围绕这个现实而建立起来的。无论您是否意识到，HTTP都是您每天都在使用的东西。通过Symfony，你将学习如何掌握它。

第一步：客户端发出请求¶

网络上的每个对话都以一个请求开始。 该请求是由被称为HTTP的特殊格式的客户端（例如浏览器，智能手机应用程序等）创建的文本消息。 客户端将该请求发送到服务器，然后等待响应。

看看浏览器和网络XKCD服务器之间的交互（请求）的第一部分：

在HTTP中，这个HTTP请求实际上看起来像这样：

GET / HTTP/1.1
Host: xkcd.com
Accept: text/html
User-Agent: Mozilla/5.0 (Macintosh)

这个简单的消息传达了客户端正在请求的资源的所有必要信息。 HTTP请求的第一行是最重要的，它包含两件事：URI和HTTP方法。

URI（例如/，/contact等）是标识客户想要的资源的唯一地址或位置。 HTTP方法（例如GET）定义了您想要对资源执行的操作。 HTTP方法是请求的动词并定义了可以对资源采取行动的几种常用方式：

GET 从服务器检索资源
POST 在服务器上创建一个资源
PUT 更新服务器上的资源
DELETE 从服务器上删除资源

考虑到这一点，您可以想象HTTP请求可能看起来像删除特定的博客条目，例如：

DELETE /blog/15 HTTP/1.1

注解

实际上HTTP规范定义了9个HTTP方法，但其中许多方法没有被广泛使用或支持。 实际上，许多现代浏览器甚至不支持PUT和DELETE方法。

除了第一行之外，HTTP请求总是包含称为请求标头的其他信息行。 标题可以提供广泛的信息，例如请求的主机，客户端接受的响应格式（Accept）和客户端用来发出请求的应用程序（用户代理）。 还有许多其他标题可以在维基百科的HTTP标题栏文章列表中找到。

第二步：服务端返回响应¶

一旦服务器收到请求，它就会确切地知道客户端需要哪些资源（通过URI）以及客户端想要如何使用该资源（通过方法）。 例如，在GET请求的情况下，服务器准备资源并将其返回到HTTP响应中。 考虑xkcd web服务器的响应：

翻译成HTTP，发送回浏览器的响应将如下所示：

HTTP/1.1 200 OK
Date: Sat, 02 Apr 2011 21:05:05 GMT
Server: lighttpd/1.4.19
Content-Type: text/html

<html>
  <!-- ... HTML for the xkcd comic -->
</html>

HTTP响应包含请求的资源（本例中为HTML内容）以及有关响应的其他信息。 第一行非常重要，包含HTTP响应状态码（在这种情况下为200）。 状态码将请求的总体结果传回客户端。 请求是否成功？ 有错误吗？ 不同的状态代码表示成功，错误或客户需要做某些事情（例如重定向到另一页）。 完整列表可以在Wikipedia的HTTP状态代码列表中找到。

与请求一样，HTTP响应包含额外的信息，即HTTP头。 例如，一个重要的HTTP响应头是Content-Type。 相同资源的主体可以以多种不同格式返回，如HTML，XML或JSON，而Content-Type标题使用Internet媒体类型（如text/html）告诉客户端正在返回哪种格式。 常见媒体类型列表可以在Wikipedia的常见媒体类型文章列表中找到。

还有许多其他标题，其中一些非常强大。 例如，某些标题可用于创建强大的缓存系统。

Requests, Responses和Web开发¶

这种请求 - 响应对话是驱动网络上所有通信的基本过程。 作为这个过程的重要和强大，它必须是非常简单的。

最重要的事实是：无论您使用哪种语言，您构建的应用程序类型（Web，移动，JSON API）还是您遵循的开发哲学，应用程序的最终目标都是了解每个请求并创建并 返回适当的响应。

Symfony的架构符合这一现实。

小技巧

要了解有关HTTP规范的更多信息，请阅读原始HTTP 1.1 RFC或HTTP Bis，这是一项积极努力来澄清原始规范。 在Firefox的Live HTTP Headers扩展是浏览时检查请求和响应标头的一个非常好的工具。

PHP里的请求和响应¶

那么，如何在使用PHP时与“请求”进行交互并创建“响应”？ 实际上，PHP从整个过程中抽象出一点：

$uri = $_SERVER['REQUEST_URI'];
$foo = $_GET['foo'];

header('Content-Type: text/html');
echo 'The URI requested is: '.$uri;
echo 'The value of the "foo" parameter is: '.$foo;

听起来很奇怪，这个小应用程序实际上是从HTTP请求中获取信息并使用它来创建HTTP响应。 PHP不是解析原始HTTP请求消息，而是准备包含请求中所有信息的超全局变量，例如$_SERVER和$_GET。 类似地，您可以使用header()函数创建响应头并简单地打印出将成为响应消息的内容部分的实际内容，而不是返回HTTP格式的文本响应。 PHP将创建一个真正的HTTP响应并将其返回给客户端： 

HTTP/1.1 200 OK
Date: Sat, 03 Apr 2011 02:14:33 GMT
Server: Apache/2.2.17 (Unix)
Content-Type: text/html

The URI requested is: /testing?foo=symfony
The value of the "foo" parameter is: symfony

Symfony里的请求和响应¶

Symfony通过两个类提供了一种替代原始PHP方法的方法，它允许您以更简单的方式与HTTP请求和响应进行交互。Request类是HTTP请求消息的简单面向对象表示。 有了它，您可以轻松掌握所有请求信息： 

use Symfony\Component\HttpFoundation\Request;

$request = Request::createFromGlobals();

// the URI being requested (e.g. /about) minus any query parameters
$request->getPathInfo();

// retrieve GET and POST variables respectively
$request->query->get('foo');
$request->request->get('bar', 'default value if bar does not exist');

// retrieve SERVER variables
$request->server->get('HTTP_HOST');

// retrieves an instance of UploadedFile identified by foo
$request->files->get('foo');

// retrieve a COOKIE value
$request->cookies->get('PHPSESSID');

// retrieve an HTTP request header, with normalized, lowercase keys
$request->headers->get('host');
$request->headers->get('content_type');

$request->getMethod();    // GET, POST, PUT, DELETE, HEAD
$request->getLanguages(); // an array of languages the client accepts

作为奖励，Request类在背后做了很多工作，你永远不需要担心。 例如，isSecure()方法检查PHP中的三个不同值，它们可以指示用户是否通过安全连接（即HTTPS）进行连接。

ParameterBags和请求属性

如上所示，$_GET和$_POST变量可以分别通过公共查询和请求属性访问。 每个对象都是一个ParameterBag对象，它具有 get(), has(), all()等多种方法。 事实上，前面例子中使用的每个公共属性都是ParameterBag的一些实例。

Request类还具有公共属属性，该 属性包含与应用程序内部工作方式有关的特殊数据。 对于Symfony框架，这些属性包含匹配路由返回的值，如_controller，id（如果您有{id}通配符），甚至匹配路由的名称（_route）。 attributes属性完全存在，您可以在其中准备和存储有关请求的特定于上下文的信息。

Symfony还提供了一个Response类：一个HTTP响应消息的简单PHP表示。 这允许您的应用程序使用面向对象的接口来构造需要返回给客户端的响应： 

use Symfony\Component\HttpFoundation\Response;

$response = new Response();

$response->setContent('<html><body><h1>Hello world!</h1></body></html>');
$response->setStatusCode(Response::HTTP_OK);
$response->headers->set('Content-Type', 'text/html');

// prints the HTTP headers followed by the content
$response->send();

如果Symfony没有提供任何其他的东西，那么您将拥有一个轻松访问请求信息的工具包和一个用于创建响应的面向对象的接口。 即使您在了解Symfony中的许多强大功能时，请记住，应用程序的目标始终是解释请求并根据您的应用程序逻辑创建适当的响应。

小技巧

Request和Response类是Symfony中包含的名为HttpFoundation的独立组件的一部分。 该组件可完全独立于Symfony使用，并提供用于处理会话和文件上传的类。

从Request到Response的旅程¶

与HTTP本身一样，Request和Response对象非常简单。 构建应用程序的难点在于编写之间的内容。 换句话说，真正的工作是编写解释请求信息并创建响应的代码。

您的应用程序可能会执行许多操作，例如发送电子邮件，处理表单提交，将内容保存到数据库，呈现HTML页面以及保护内容安全。 你如何管理所有这些，并仍然保持你的代码组织和可维护？

Symfony的创建是为了解决这些问题，所以你不必这样做。

The Front Controller¶

传统上，应用程序的建立使网站的每个“页面”都是自己的物理文件：

index.php
contact.php
blog.php

这种方法存在几个问题，包括URL的灵活性（如果您想将blog.php更改为news.php而不破坏所有链接？）以及每个文件都必须手动包含一些核心文件集的事实 因此安全性，数据库连接和站点的“外观”可以保持一致。

一个更好的解决方案是使用前端控制器：一个PHP文件处理进入应用程序的每个请求。 例如：

/index.php 执行 index.php
/index.php/contact 执行 index.php
/index.php/blog 执行 index.php

小技巧

使用Apache的mod_rewrite（或与其他Web服务器相同），URL可以轻松清理为/, /contact and /blog。

现在，每个请求都以完全相同的方式处理。 代替单个URL执行不同的PHP文件，前端控制器总是被执行，而不同的URL到应用程序的不同部分的路由在内部完成。 这解决了原始方法的两个问题。 几乎所有的现代网络应用都可以做到这一点 - 包括WordPress等应用。

保持条理清晰¶

在你的前端控制器里面，你必须弄清楚哪些代码应该被执行以及返回的内容应该是什么。 为了解决这个问题，您需要检查传入的URI并根据该值执行代码的不同部分。 这会很快变得丑陋： 

// index.php
use Symfony\Component\HttpFoundation\Request;
use Symfony\Component\HttpFoundation\Response;

$request = Request::createFromGlobals();
$path = $request->getPathInfo(); // the URI path being requested

if (in_array($path, array('', '/'))) {
    $response = new Response('Welcome to the homepage.');
} elseif ('/contact' === $path) {
    $response = new Response('Contact us');
} else {
    $response = new Response('Page not found.', Response::HTTP_NOT_FOUND);
}
$response->send();

解决这个问题可能很困难。 幸运的是，这正是Symfony设计的目的。

Symfony应用的运行流程¶

当你让Symfony处理每个请求时，生活变得更容易。 Symfony对每个请求都遵循相同的简单模式：

传入的请求由路由解释并传递给返回Response对象的控制器函数。

您网站的每个“页面”都在路由配置文件中定义，该文件将不同的URL映射到不同的PHP功能。 每个PHP函数（称为控制器）的工作是使用请求中的信息以及Symfony提供的许多其他工具来创建和返回Response对象。 换句话说，控制器就是您的代码所在的地方：它是您解释请求并创建响应的地方。

这很简单！ 回顾：

  • 每个请求执行一个前端控制器文件;
  • 路由系统根据您创建的请求和路由配置中的信息确定应该执行哪个PHP函数;
  • 执行正确的PHP函数，您的代码创建并返回相应的Response对象。

Symfony请求处理示例¶

没有深入细节，这里就是这个过程。 假设你想添加一个/contact页面到你的Symfony应用程序。 首先，开始为您的路由配置文件添加一个/contact的条目：

  • YAML 
    # app/config/routing.yml
contact:
    path:     /contact
    defaults: { _controller: AppBundle:Main:contact }
  • XML 
    <!-- app/config/routing.xml -->
<?xml version="1.0" encoding="UTF-8" ?>
<routes xmlns="http://symfony.com/schema/routing"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="http://symfony.com/schema/routing
        http://symfony.com/schema/routing/routing-1.0.xsd">

    <route id="contact" path="/contact">
        <default key="_controller">AppBundle:Main:contact</default>
    </route>
</routes>
  • PHP 
    // app/config/routing.php
use Symfony\Component\Routing\Route;
use Symfony\Component\Routing\RouteCollection;

$collection = new RouteCollection();
$collection->add('contact', new Route('/contact', array(
    '_controller' => 'AppBundle:Main:contact',
)));

return $collection;

当有人访问/contact页面时，该路由匹配，并执行指定的控制器。 正如你将在路由章节中学到的，AppBundle:Main:contact string是一个简短的语法，指向一个名为MainController的类中的特定PHP方法contactAction： 

// src/AppBundle/Controller/MainController.php
namespace AppBundle\Controller;

use Symfony\Component\HttpFoundation\Response;

class MainController
{
    public function contactAction()
    {
        return new Response('<h1>Contact us!</h1>');
    }
}

在这个非常简单的例子中，控制器只需用HTML <h1>Contact us!</h1>创建一个Response对象。 在控制器章节 chapter中，您将学习控制器如何呈现模板，从而允许您的“演示文稿”代码（即实际写出HTML的任何内容）位于单独的模板文件中。 这使得控制器只需要担心困难的事情：与数据库交互，处理提交的数据或发送电子邮件。

Symfony：专注项目本身，而不是工具¶

您现在知道任何应用程序的目标都是解释每个传入的请求并创建适当的响应。 随着应用程序的增长，保持代码的组织和维护变得更加困难。 不变的是，同样复杂的任务不断出现：持久化事物到数据库，渲染和重用模板，处理表单提交，发送电子邮件，验证用户输入和处理安全性。

好消息是这些问题都不是独一无二的。 Symfony提供了一个完整的工具框架，允许您构建应用程序，而不是工具。 使用Symfony时，您不需要强加任何东西：您可以自由使用完整的Symfony框架，或者仅使用一块Symfony。

独立的工具箱: Symfony 组件¶

那么Symfony是什么？ 首先，Symfony是超过二十个独立库的集合，可以在任何PHP项目中使用。 这些名为Symfony组件的库包含几乎适用于任何情况的东西，无论您的项目是如何开发的。 仅举几例：

HttpFoundation
包含请求和响应类以及处理会话和文件上传的其他类。
Routing
功能强大且快速的路由系统，允许您将特定URI（例如/contact）映射到有关如何处理该请求的某些信息（例如，执行contactAction()方法）。
Form
用于创建表单和处理表单提交的功能齐全且灵活的框架。
Validator
用于创建关于数据的规则然后验证用户提交的数据是否遵循这些规则的系统。
Templating
用于呈现模板，处理模板继承（即模板用布局装饰）和执行其他常见模板任务的工具包。
Security
一个强大的库，用于处理应用程序内的所有类型的安全性。
Translation
用于在应用程序中转换字符串的框架。

无论您是否使用Symfony框架，这些组件中的每一个都是分离的，并可用于任何PHP项目。 如果需要，每个零件都可以使用并在必要时更换。

全方位解决方案: Symfony 框架¶

那么，什么是Symfony框架？ Symfony框架是一个PHP库，它可以完成两个不同的任务：

  1. 提供选择的组件（即Symfony组件）和第三方库（例如用于发送电子邮件的 Swift Mailer）;
  2. 提供合理的配置和一个将所有这些碎片连接在一起的“glue”库。

该框架的目标是集成许多独立工具，以便为开发人员提供一致的体验。 即使框架本身也是一个可以完全配置或替换的Symfony包（即插件）。

Symfony提供了一套强大的工具，用于快速开发Web应用程序，而不会对应用程序施加压力。 普通用户可以通过使用Symfony发行版来快速开始开发，该发行版提供了具有合理默认值的项目框架。 对于更高级的用户来说，上不封顶。
