创建页面¶

在Symfony中创建一个新页面是一个简单的两步过程:

  • 创建一个路由: 路由定义了页面的URL（例如/about），并指定了一个控制器（这是一个PHP函数），当一个传入请求的URL匹配路由路径时，Symfony应该执行该控制器
  • 创建控制器：控制器是一个PHP函数，它接收传入的请求并将其转换为返回给用户的Symfony Response对象。

这个简单的方法很漂亮，因为它与Web的工作方式相匹配。 Web上的每个交互都由HTTP请求启动。 您的应用程序的工作仅仅是解释请求并返回适当的HTTP响应。

Symfony遵循这一理念，为您提供工具和约定，让您的应用随着用户的增长和复杂性而组织。

运行环境 & Front Controllers¶

每个Symfony应用程序都在一个环境中运行。 环境是一组特定的配置和加载的包，由一个字符串表示。 通过在不同的环境中运行应用程序，可以使用不同的配置运行相同的应用程序 Symfony提供了三种定义的环境 - dev, test 和 prod - 但您也可以创建自己的环境。

通过允许单个应用程序具有用于调试的开发环境和针对速度优化的生产环境，环境非常有用。 您也可以根据选定的环境加载特定的软件包。 例如，Symfony附带WebProfilerBundle（如下所述），仅在开发和测试环境中启用。

Symfony带有两个网络可访问的前端控制器：app_dev.php提供了开发环境，而app.php提供了生产环境。 所有对Symfony的Web访问通常都是通过这些前端控制器之一。 （测试环境通常只在运行单元测试时使用，因此没有专用的前端控制器，控制台工具还提供了可用于任何环境的前端控制器。）

当前端控制器初始化内核时，它提供两个参数：环境，以及内核是否应以调试模式运行。 为了让您的应用程序更快响应，Symfony在app/cache/目录下维护一个缓存。 当启用调试模式（例如app_dev.php默认）时，只要您更改任何代码或配置，就会自动刷新此缓存。 在调试模式下运行时，Symfony运行速度较慢，但您的更改反映出来而无需手动清除缓存。

“随机数”页面示例¶

在本章中，您将开发一个可以生成随机数的应用程序。 完成后，用户将能够获得1到由URL设置的上限的随机数： 

http://localhost/app_dev.php/random/100

实际上，你可以用任何其他号码代替100来产生高达上限的数字。 要创建页面，请执行简单的两步过程。

注解

本教程假设您已经下载了Symfony并配置了您的网络服务器。 上面的URL假定localhost指向新的Symfony项目的web目录。 有关此过程的详细信息，请参阅您正在使用的Web服务器上的文档。 以下是您可能正在使用的Web服务器的一些相关文档页面：

  • 对于Apache HTTP Server，请参阅Apache的DirectoryIndex文档
  • 对于Nginx，请参阅Nginx HttpCoreModule位置文档

在您开始之前请先创建Bundle¶

在开始之前，您需要创建一个bundle。 在Symfony中，一个bundle就像一个插件，除了应用程序中的所有代码将存在于一个包中。

一个bundle只不过是一个包含与特定功能相关的所有东西的目录，包括PHP类，配置，甚至是样式表和JavaScript文件（请参阅Bundle系统）。

根据您安装Symfony的方式，您可能已经有一个名为AcmeDemoBundle的包。 浏览项目的src/目录，并检查Acme/目录内是否有DemoBundle/目录。 如果这些目录已经存在，则跳过本节的其余部分，直接创建router。

要创建名为AcmeDemoBundle的包（您将在本章中构建的游戏包），请运行以下命令并按照屏幕上的说明操作（使用所有默认选项）： 

$ php app/console generate:bundle --namespace=Acme/DemoBundle --format=yml

在幕后，在src/Acme/DemoBundle中为该包创建一个目录。 一行也会自动添加到app/AppKernel.php文件中，以便该软件包在内核中注册： 

// app/AppKernel.php
public function registerBundles()
{
    $bundles = array(
        // ...
        new Acme\DemoBundle\AcmeDemoBundle(),
    );
    // ...

    return $bundles;
}

现在您已经安装了软件包，您可以开始在软件包中构建应用程序。

第一步：创建Route¶

默认情况下，Symfony应用程序中的路由配置文件位于app/config/routing.yml。 像Symfony中的所有配置一样，您也可以选择使用XML或PHP开箱即可配置路由。

如果您查看主路由文件，您会看到Symfony在生成AcmeDemoBundle时已经添加了一个条目：

  • YAML 
    # app/config/routing.yml
acme_website:
    resource: "@AcmeDemoBundle/Resources/config/routing.yml"
    prefix:   /
  • XML 
    <!-- app/config/routing.xml -->
<?xml version="1.0" encoding="UTF-8" ?>
<routes xmlns="http://symfony.com/schema/routing"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="http://symfony.com/schema/routing
        http://symfony.com/schema/routing/routing-1.0.xsd">

    <import
        resource="@AcmeDemoBundle/Resources/config/routing.xml"
        prefix="/" />
</routes>
  • PHP 
    // app/config/routing.php
use Symfony\Component\Routing\RouteCollection;

$acmeDemo = $loader->import('@AcmeDemoBundle/Resources/config/routing.php');
$acmeDemo->addPrefix('/');

$collection = new RouteCollection();
$collection->addCollection($acmeDemo);

return $collection;

这个条目非常基础：它告诉Symfony从位于AcmeDemoBundle内的Resources/config/routing.yml（分别为XML和PHP代码示例中的routing.xml或routing.php）文件加载路由配置。 这意味着您将路由配置直接放置在app/config/routing.yml中，或者在整个应用程序中组织您的路由，并从这里导入它们。

注解

您不限于加载格式相同的路由配置。 例如，您也可以在XML配置中加载YAML文件，反之亦然。

现在已经导入了bundle中的routing.yml文件，添加了新的路由来定义您即将创建的页面的URL：

  • YAML 
    # src/Acme/DemoBundle/Resources/config/routing.yml
random:
    path:     /random/{limit}
    defaults: { _controller: AcmeDemoBundle:Random:index }
  • XML 
    <!-- src/Acme/DemoBundle/Resources/config/routing.xml -->
<?xml version="1.0" encoding="UTF-8" ?>
<routes xmlns="http://symfony.com/schema/routing"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xsi:schemaLocation="http://symfony.com/schema/routing
        http://symfony.com/schema/routing/routing-1.0.xsd">

    <route id="random" path="/random/{limit}">
        <default key="_controller">AcmeDemoBundle:Random:index</default>
    </route>
</routes>
  • PHP 
    // src/Acme/DemoBundle/Resources/config/routing.php
use Symfony\Component\Routing\RouteCollection;
use Symfony\Component\Routing\Route;

$collection = new RouteCollection();
$collection->add('random', new Route('/random/{limit}', array(
    '_controller' => 'AcmeDemoBundle:Random:index',
)));

return $collection;

路由由两个基本部分组成：path（即此路由将匹配的URL）和defaults数组（指定应执行的控制器）。 路径中的占位符语法（{limit}）是通配符。 这意味着 /random/10，/random/327或任何其他类似的URL将匹配此路线。 {limit}占位符参数也将传递给控制器，以便您可以使用它的值生成适当的随机数。

注解

路由系统具有更多强大的功能，可以在您的应用程序中创建灵活而强大的URL结构。 有关更多详细信息，请参阅有关Routing.的所有章节。

第二部：创建Controller¶

当应用程序处理诸如/random/10之类的URL时，random路由匹配并且AcmeDemoBundle:Random:index控制器由框架执行。 页面创建过程的第二步是创建该控制器。

控制器 -AcmeDemoBundle:Random:index是控制器的逻辑名称，它映射到名为Acme\DemoBundle\Controller\RandomController的PHP类的indexAction方法。 首先在你的AcmeDemoBundle中创建这个文件： 

// src/Acme/DemoBundle/Controller/RandomController.php
namespace Acme\DemoBundle\Controller;

class RandomController
{
}

实际上，控制器只不过是您创建和Symfony执行的PHP方法。 这是您的代码使用请求中的信息构建和准备请求的资源的位置。 除了一些先进的情况，控制器的最终产品总是相同的：一个SymfonyResponse对象。

创建在匹配随random由时Symfony将执行的indexAction方法： 

// src/Acme/DemoBundle/Controller/RandomController.php
namespace Acme\DemoBundle\Controller;

use Symfony\Component\HttpFoundation\Response;

class RandomController
{
    public function indexAction($limit)
    {
        return new Response(
            '<html><body>Number: '.rand(1, $limit).'</body></html>'
        );
    }
}

控制器很简单：它创建一个新的Response对象，其第一个参数是应该在响应中使用的内容（本例中是一个小的HTML页面）。

恭喜！ 创建只有一个路由和一个控制器后，你已经有了一个功能齐全的页面！ 如果你已经正确设置了一切，你的应用程序应该为你生成一个随机数字：

http://localhost/app_dev.php/random/10

小技巧

您也可以通过访问以下网址在“产品”环境中查看您的应用程序： 

http://localhost/app.php/random/10

如果出现错误，可能是因为您需要通过运行以清除缓存：

$ php app/console cache:clear --env=prod --no-debug

该过程中的可选但常见的第三步是创建模板。

注解

控制器是您的代码的主要入口点，也是创建页面时的关键因素。 更多的信息可以在Controller 章节找到。

通过模板，您可以将所有演示文稿代码（例如HTML）移动到单独的文件中，并重新使用页面布局的不同部分。 而不是在控制器中写入HTML，而是渲染一个模板：

// src/Acme/DemoBundle/Controller/RandomController.php
namespace Acme\DemoBundle\Controller;

use Symfony\Bundle\FrameworkBundle\Controller\Controller;

class RandomController extends Controller
{
    public function indexAction($limit)
    {
        $number = rand(1, $limit);

        return $this->render(
            'AcmeDemoBundle:Random:index.html.twig',
            array('number' => $number)
        );

        // render a PHP template instead
        // return $this->render(
        //     'AcmeDemoBundle:Random:index.html.php',
        //     array('number' => $number)
        // );
    }
}

注解

为了使用render()方法，您的控制器必须扩展Controller类，它为控制器内常见的任务添加快捷方式。 在上面的例子中，通过在第4行添加use语句，然后在第6行扩展Controller来完成此操作。

render()方法创建一个填充了给定渲染模板内容的Response对象。 像任何其他控制器一样，您最终将返回该Response对象。

注意渲染模板有两个不同的例子。 默认情况下，Symfony支持两种不同的模板语言：经典的PHP模板和简洁但功能强大的Twig模板。 不要惊慌 - 你可以在同一个项目中自由选择，甚至可以选择两者。

控制器呈现AcmeDemoBundle:Random:index.html.twig模板，该模板使用以下命名约定：

BundleName:ControllerName:TemplateName

这是模板的逻辑名称，使用以下约定将其映射到物理位置。

/path/to/BundleName/Resources/views/ControllerName/TemplateName

在这种情况下，AcmeDemoBundle是包名称，Random是控制器，index.html.twig是模板：

    <!-- src/Acme/DemoBundle/Resources/views/Random/index.html.php -->
<?php $view->extend('::base.html.php') ?>

Number: <?php echo $view->escape($number) ?>

逐行浏览Twig模板：

  • 第2行：extends令牌定义了一个父模板。 该模板明确定义了将放置在其中的布局文件。
  • 第4行： block表示里面的所有内容都应放在称为body的块内。 如您所见，父模板（base.html.twig）负责最终呈现名为body的块。

父模板::base.html.twig缺少其名称的BundleName和ControllerName部分（因此在开头处因此是双冒号（::）。 这意味着该模板位于软件包外部和app程序目录中：

  • Twig 
    {# app/Resources/views/base.html.twig #}
<!DOCTYPE html>
<html>
    <head>
        <meta http-equiv="Content-Type" content="text/html; charset=utf-8" />
        <title>{% block title %}Welcome!{% endblock %}</title>
        {% block stylesheets %}{% endblock %}
        <link rel="shortcut icon" href="{{ asset('favicon.ico') }}" />
    </head>
    <body>
        {% block body %}{% endblock %}
        {% block javascripts %}{% endblock %}
    </body>
</html>
  • PHP 
    <!-- app/Resources/views/base.html.php -->
<!DOCTYPE html>
<html>
    <head>
        <meta http-equiv="Content-Type" content="text/html; charset=utf-8" />
        <title><?php $view['slots']->output('title', 'Welcome!') ?></title>
        <?php $view['slots']->output('stylesheets') ?>
        <link rel="shortcut icon"
            href="<?php echo $view['assets']->getUrl('favicon.ico') ?>" />
    </head>
    <body>
        <?php $view['slots']->output('_content') ?>
        <?php $view['slots']->output('javascripts') ?>
    </body>
</html>

基本模板文件定义HTML布局并呈现您在index.html.twig 模板中定义的body块。 它还呈现一个标题栏，您可以选择在index.html.twig模板中定义该标题栏。 由于您没有在子模板中定义标题栏，因此它默认为“欢迎！”。

模板是呈现和组织页面内容的强大方式。 模板可以呈现任何内容，从HTML标记到CSS代码或控制器可能需要返回的任何其他内容。

在处理请求的生命周期中，模板引擎只是一个可选工具。 回想一下，每个控制器的目标是返回一个Response对象。 模板是一个功能强大但可选的工具，用于为该Response对象创建内容。

目录结构¶

在几个简短的部分之后，您已经了解了在Symfony中创建和渲染页面的理念。您也已经开始了解Symfony项目的结构和组织方式。到本节结束时，您将知道在哪里可以找到并放置不同类型的文件以及为什么。

虽然完全灵活，但默认情况下，每个Symfony应用程序具有相同的基本和推荐的目录结构：

app/
该目录包含应用程序配置。
src/
所有项目的PHP代码都存储在这个目录下。
vendor/
任何vender库都按照惯例放置在这里。
web/
这是Web根目录并包含任何可公开访问的文件。

参见

您可以轻松覆盖默认的目录结构。 有关更多信息，请参阅如何重载Symfony默认的目录结构。

Web目录¶

Web根目录是所有公共和静态文件的主页，包括图像，样式表和JavaScript文件。 这也是每个前端控制器所在的地方： 

// web/app.php
require_once __DIR__.'/../app/bootstrap.php.cache';
require_once __DIR__.'/../app/AppKernel.php';

use Symfony\Component\HttpFoundation\Request;

$kernel = new AppKernel('prod', false);
$kernel->loadClassCache();
$kernel->handle(Request::createFromGlobals())->send();

前端控制器文件（本例中为app.php）是使用Symfony应用程序时执行的实际PHP文件，其作业是使用Kernel类AppKernel来引导应用程序。

小技巧

拥有前端控制器意味着与典型的平面PHP应用程序中使用的不同且更灵活的URL。 使用前端控制器时，URL格式如下：

http://localhost/app.php/random/10

执行前端控制器app.php，并使用路由配置在内部路由“internal：”URL /random/10。 通过使用Apache mod_rewrite规则，您可以强制执行app.php文件，而无需在URL中指定它： 

http://localhost/random/10

尽管前端控制器对于处理每个请求都很重要，但您很少需要修改甚至考虑它们。 他们将在`Environments`_部分简要介绍。

应用（ app ）目录¶

正如您在前端控制器中看到的，AppKernel类是应用程序的主要入口点，并负责所有配置。 因此，它存储在app/目录中。

这个类必须实现两个方法来定义Symfony需要了解的应用程序的所有信息。 开始时你甚至不需要担心这些方法 - Symfony以合理的默认值填充它们。

registerBundles()
返回运行应用程序所需的所有bundle的数组（请参阅Bundle系统）。
registerContainerConfiguration()
Loads the main application configuration resource file (see the section). 加载主应用程序配置资源文件（请参阅“`应用程序配置`”部分）。

在日常开发中，您大多会使用app/目录来修改app/config/目录中的配置和路由文件（请参阅“应用程序配置”）。 它还包含应用程序缓存目录（app/cache），日志目录（app/logs）和应用程序级资源文件（如模板（app/Resources））的目录。 您将在后面的章节中了解更多关于这些目录的信息。

自动加载

当Symfony加载时，包含一个特殊文件 - vendor/autoload.php。 该文件由Composer创建，可自动加载生活在src/文件夹中的所有应用程序文件以及composer.json文件中提及的所有第三方库。

由于自动加载器，您永远不必担心使用include或require语句。 相反，Composer使用类的命名空间来确定它的位置，并在您需要类的时候自动为您包含该文件。

自动加载器已经配置为在src/目录中查找任何PHP类。 为了自动加载，文件的类名和路径必须遵循相同的模式： 

Class Name:
    Acme\DemoBundle\Controller\RandomController
Path:
    src/Acme/DemoBundle/Controller/RandomController.php

代码（ src ）目录¶

简而言之，src/目录包含驱动应用程序的所有实际代码（PHP代码，模板，配置文件，样式表等）。 开发时，绝大多数工作将在您在此目录中创建的一个或多个捆绑包中完成。

但是，bundle到底是什么？

Bundle系统¶

软件包类似于其他软件中的插件，但更好。 关键的区别在于，Symfony中的所有东西都是捆绑包，包括核心框架功能和为您的应用程序编写的代码。 Bundle是Symfony的一等公民。 这使您可以灵活地使用第三方软件包中打包的预构建功能或分发自己的软件包。 它可以轻松挑选和选择在应用程序中启用哪些功能，并按照您的需要进行优化。

注解

虽然您将在此学习基础知识，但整本食谱条目都专门针对bundles的组织和最佳实践。

一个bundle只是一个目录中的结构化文件集，可以实现单个功能。 您可以创建一个BlogBundle，一个ForumBundle或一个用于用户管理的包（其中许多已经作为开源捆绑包存在）。 每个目录都包含与该功能相关的所有内容，包括PHP文件，模板，样式表，JavaScript，测试和其他任何内容。 一个功能的每个方面都存在于一个包中，每个功能都在一个包中。

应用程序由AppKernel 类的registerBundles()方法中定义的bundle组成： 

// app/AppKernel.php
public function registerBundles()
{
    $bundles = array(
        new Symfony\Bundle\FrameworkBundle\FrameworkBundle(),
        new Symfony\Bundle\SecurityBundle\SecurityBundle(),
        new Symfony\Bundle\TwigBundle\TwigBundle(),
        new Symfony\Bundle\MonologBundle\MonologBundle(),
        new Symfony\Bundle\SwiftmailerBundle\SwiftmailerBundle(),
        new Symfony\Bundle\DoctrineBundle\DoctrineBundle(),
        new Symfony\Bundle\AsseticBundle\AsseticBundle(),
        new Sensio\Bundle\FrameworkExtraBundle\SensioFrameworkExtraBundle(),
    );

    if (in_array($this->getEnvironment(), array('dev', 'test'))) {
        $bundles[] = new Acme\DemoBundle\AcmeDemoBundle();
        $bundles[] = new Symfony\Bundle\WebProfilerBundle\WebProfilerBundle();
        $bundles[] = new Sensio\Bundle\DistributionBundle\SensioDistributionBundle();
        $bundles[] = new Sensio\Bundle\GeneratorBundle\SensioGeneratorBundle();
    }

    return $bundles;
}

使用registerBundles()方法，您可以完全控制应用程序使用哪些捆绑包（包括Symfony核心捆绑包）。

小技巧

只要可以自动加载（通过在app/autoload.php中配置的自动加载器），捆绑包可以在任何地方生活。

创建Bundle¶

Symfony标准版附带了一个方便的任务，为您创建一个功能齐全的软件包。 当然，手工创建捆绑包也很容易。.

为了向您展示捆绑系统的简单性，创建一个名为AcmeTestBundle的新捆绑包并启用它。

小技巧

Acme部分只是一个虚拟名称，应该由代表您或您的组织的某个“供应商”名称替换（例如，某家名为ABC的ABCTestBundle）。

首先创建一个src/Acme/TestBundle/目录并添加一个名为AcmeTestBundle.php的新文件： 

// src/Acme/TestBundle/AcmeTestBundle.php
namespace Acme\TestBundle;

use Symfony\Component\HttpKernel\Bundle\Bundle;

class AcmeTestBundle extends Bundle
{
}

小技巧

AcmeTestBundle的名称遵循标准的Bundle命名约定。 您也可以选择通过命名此TestBundle类（并命名文件TestBundle.php）来简化TestBundle的名称。

这个空的类是您创建新bundle所需的唯一部分。 虽然通常是空的，但这个类很强大，可以用来定制bundle的行为。

现在您已经创建了该bundle包，请通过AppKernel类启用它： 

// app/AppKernel.php
public function registerBundles()
{
    $bundles = array(
        // ...
        // register your bundle
        new Acme\TestBundle\AcmeTestBundle(),
    );
    // ...

    return $bundles;
}

虽然它还没有做任何事情，但AcmeTestBundle现在已经可以使用了。

就像这样简单，Symfony还提供了一个命令行界面来生成一个基本的包骨架：

$ php app/console generate:bundle --namespace=Acme/TestBundle

包骨架通过可自定义的基本控制器，模板和路由资源生成。 稍后您将学习更多关于Symfony的命令行工具。

小技巧

无论何时创建新捆绑包或使用第三方捆绑包，始终确保该捆绑包已在registerBundles()中启用。 使用generate:bundle命令时，这是为您完成的。

Bundle目录结构¶

一个包的目录结构简单而灵活。 默认情况下，捆绑系统遵循一系列约定来帮助保持所有Symfony捆绑包之间的代码一致。 看一看AcmeDemoBundle，因为它包含了一些最常见的包：

Controller/
包含该包的控制器（例如RandomController.php）。
DependencyInjection/
持有某些依赖注入扩展类，可能导入服务配置，注册编译器通过或更多（此目录不是必需的）。
Resources/config/
房屋配置，包括路由配置（例如routing.yml）。
Resources/views/
保存按控制器名称组织的模板（例如，Hello/index.html.twig）。
Resources/public/
包含网络资产（图片，样式表等），并通过assets:install命令复制或符号链接到项目web/目录。
Tests/
持有该包的所有测试。

捆绑包可以与其实现的特征一样小或大。 它仅包含您需要的文件，而不包含其他文件。

在阅读本书时，您将学习如何将对象保存到数据库，创建和验证表单，为应用程序创建翻译，编写测试等等。 这些中的每一个在捆绑中都有自己的位置和角色。

应用配置¶

应用程序由一系列代表应用程序的所有功能和功能的软件包组成。 每个软件包都可以通过用YAML，XML或PHP编写的配置文件进行定制。 默认情况下，主配置文件位于app/config/目录中，根据您喜欢的格式调用config.yml，config.xml或config.php：

  • YAML 
    # app/config/config.yml
imports:
    - { resource: parameters.yml }
    - { resource: security.yml }

framework:
    secret:          "%secret%"
    router:          { resource: "%kernel.root_dir%/config/routing.yml" }
    # ...

# Twig Configuration
twig:
    debug:            "%kernel.debug%"
    strict_variables: "%kernel.debug%"

# ...
  • XML 
    <!-- app/config/config.xml -->
<?xml version="1.0" encoding="UTF-8" ?>
<container xmlns="http://symfony.com/schema/dic/services"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xmlns:framework="http://symfony.com/schema/dic/symfony"
    xmlns:twig="http://symfony.com/schema/dic/twig"
    xsi:schemaLocation="http://symfony.com/schema/dic/services
        http://symfony.com/schema/dic/services/services-1.0.xsd
        http://symfony.com/schema/dic/symfony
        http://symfony.com/schema/dic/symfony/symfony-1.0.xsd
        http://symfony.com/schema/dic/twig
        http://symfony.com/schema/dic/twig/twig-1.0.xsd">

    <imports>
        <import resource="parameters.yml" />
        <import resource="security.yml" />
    </imports>

    <framework:config secret="%secret%">
        <framework:router resource="%kernel.root_dir%/config/routing.xml" />
        <!-- ... -->
    </framework:config>

    <!-- Twig Configuration -->
    <twig:config debug="%kernel.debug%" strict-variables="%kernel.debug%" />

    <!-- ... -->
</container>
  • PHP 
    // app/config/config.php
$this->import('parameters.yml');
$this->import('security.yml');

$container->loadFromExtension('framework', array(
    'secret' => '%secret%',
    'router' => array(
        'resource' => '%kernel.root_dir%/config/routing.php',
    ),
    // ...
));

// Twig Configuration
$container->loadFromExtension('twig', array(
    'debug'            => '%kernel.debug%',
    'strict_variables' => '%kernel.debug%',
));

// ...

注解

您将在下一节`Environments`中准确了解如何加载每个文件/格式。

每个顶级条目（如framework或twig）都定义了特定捆绑软件的配置。 例如，framework键定义了核心Symfony FrameworkBundle的配置，并包含路由，模板和其他核心系统的配置。

现在，不要担心每个部分中的特定配置选项。 配置文件附带合理的默认值。 在阅读更多内容并探索Symfony的每个部分时，您将了解每个功能的特定配置选项。

配置格式

在整个章节中，所有配置示例将以三种格式（YAML，XML和PHP）显示。 各有其优点和缺点。 您可以选择使用哪一种：

  • YAML: 简单，干净和可读（了解更多关于YAML格式的信息）;
  • XML: 有时比YAML更强大，并支持IDE自动完成;
  • PHP: 标准配置格式非常强大但可读性差。

导入默认配置项¶

您可以使用config:dump-reference命令将YAML中软件包的默认配置转储到控制台。 以下是转储默认FrameworkBundle配置的示例： 

$ app/console config:dump-reference FrameworkBundle

扩展别名（配置密钥）也可以使用：

$ app/console config:dump-reference framework

注解

请参阅食谱文章：如何在Bundle内部载入Service配置以获取有关为您自己的包添加配置的信息。

运行环境¶

应用程序可以在各种环境中运行。 不同的环境共享相同的PHP代码（除了前端控制器），但使用不同的配置。 例如，开发环境会记录警告和错误，而产品环境只会记录错误。 一些文件在dev环境中的每个请求上被重建（为了开发者的方便），但是缓存在 prod环境中。 所有环境都在同一台机器上，并执行相同的应用程序。

Symfony项目通常以三种环境开发（dev，test和prod），但创建新环境很简单。 只需更改浏览器中的前端控制器，即可在不同的环境中查看您的应用程序。 要在dev环境中查看应用程序，请通过开发前端控制器访问应用程序： 

http://localhost/app_dev.php/random/10

如果您想查看应用程序在生产环境中的表现，请改为使用prod前端控制器： 

http://localhost/app.php/random/10

由于prod环境针对速度进行了优化， 将配置，路由和Twig模板编译为平面PHP类并进行缓存。 在prod环境中查看更改时，您需要清除这些缓存的文件并允许它们重建： 

$ php app/console cache:clear --env=prod --no-debug

注解

如果您打开web/app.php文件，您会发现它明确配置为使用prod环境： 

$kernel = new AppKernel('prod', false);

您可以通过复制此文件并将prod更改为其他值来为新环境创建新的前端控制器。

注解

测试环境在运行自动化测试时使用，不能通过浏览器直接访问。 有关更多详情，请参阅测试章节。

运行环境设置¶

AppKernel类负责实际加载您选择的配置文件： 

// app/AppKernel.php
public function registerContainerConfiguration(LoaderInterface $loader)
{
    $loader->load(
        __DIR__.'/config/config_'.$this->getEnvironment().'.yml'
    );
}

如果您更喜欢使用XML或PHP来编写配置，则您已知道.yml 扩展名可以更改为.xml或.php。 还要注意，每个环境加载它自己的配置文件。 考虑dev环境的配置文件。

  • YAML 
    # app/config/config_dev.yml
imports:
    - { resource: config.yml }

framework:
    router:   { resource: "%kernel.root_dir%/config/routing_dev.yml" }
    profiler: { only_exceptions: false }

# ...
  • XML 
    <!-- app/config/config_dev.xml -->
<?xml version="1.0" encoding="UTF-8" ?>
<container xmlns="http://symfony.com/schema/dic/services"
    xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
    xmlns:framework="http://symfony.com/schema/dic/symfony"
    xsi:schemaLocation="http://symfony.com/schema/dic/services
        http://symfony.com/schema/dic/services/services-1.0.xsd
        http://symfony.com/schema/dic/symfony
        http://symfony.com/schema/dic/symfony/symfony-1.0.xsd">

    <imports>
        <import resource="config.xml" />
    </imports>

    <framework:config>
        <framework:router resource="%kernel.root_dir%/config/routing_dev.xml" />
        <framework:profiler only-exceptions="false" />
    </framework:config>

    <!-- ... -->
</container>
  • PHP 
    // app/config/config_dev.php
$loader->import('config.php');

$container->loadFromExtension('framework', array(
    'router' => array(
        'resource' => '%kernel.root_dir%/config/routing_dev.php',
    ),
    'profiler' => array('only-exceptions' => false),
));

// ...

imports键类似于PHP include语句，并保证主配置文件（config.yml）首先被加载。 文件的其余部分调整默认配置以增加日志记录和其他有助于开发环境的设置。

prod和测试环境都遵循相同的模型：每个环境都会导入基本配置文件，然后修改其配置值以适应特定环境的需求。 这只是一个惯例，但允许您重新使用大部分配置并在环境之间自定义其中的部分内容。

总结¶

恭喜！ 你现在已经看到了Symfony的每一个基本方面，并且希望能够发现它可以变得简单和灵活。 虽然还有很多功能还未到来，但请务必牢记以下基本要点：

  • 创建页面是一个涉及路由，控制器和（可选）模板的三步过程;
  • 每个项目只包含几个主要目录：web/（web资产和前端控制器），app/（配置），src/（你的捆绑包）和vendor/（第三方代码）（还有一个bin/目录 用于帮助更新的供应商库）;
  • Symfony中的每个功能（包括Symfony框架核心）都被组织为一个包，该包是该功能的一组结构化文件;
  • 每个软件包的配置位于软件包的Resources/config目录中，可以在YAML，XML或PHP中指定;
  • 全局应用程序配置位于app/config目录中;
  • 每个环境都可以通过不同的前端控制器（例如app.php和app_dev.php）访问并加载不同的配置文件。

从这里开始，每章将向您介绍越来越多的强大工具和先进理念。 对Symfony的了解越多，您就会越感激其架构的灵活性以及它为您提供快速开发应用程序的能力。
