代码结构¶

你是我的英雄！谁会想到在前三部分之后你还会在这里呢？你的努力很快就会得到回报。 他的前三部分并没有深入研究框架的架构。 因为它让Symfony与众多架构脱颖而出，现在让我们深入到架构。

了解目录结构¶

Symfony应用程序的目录结构比较灵活，但推荐的结构如下：

app/
应用程序配置，模板和翻译。
src/
该项目的PHP代码。
vendor/
第三方依赖关系。
web/
Web根目录。

web/ 目录¶

Web根目录是所有公共和静态文件（如图像，样式表和JavaScript文件）的主目录。 It is also where each front controller lives, such as the production controller shown here: 

// web/app.php
require_once __DIR__.'/../app/bootstrap.php.cache';
require_once __DIR__.'/../app/AppKernel.php';

use Symfony\Component\HttpFoundation\Request;

$kernel = new AppKernel('prod', false);
$kernel->loadClassCache();
$request = Request::createFromGlobals();
$response = $kernel->handle($request);
$response->send();

控制器首先使用kernel类（AppKernel在这种情况下）引导应用程序。 然后，它使用PHP的全局变量创建Request对象，并将其传递给kernel。 最后一步是将kernel返回的响应内容发回给用户。

app/ 目录¶

AppKernel类是应用程序配置的主要入口点，因此它存储在app/目录中。

这个类必须实现两个方法：

registerBundles()
必须返回运行应用程序所需的所有包的数组，如下一节所述。
registerContainerConfiguration()
加载应用程序配置（稍后会详细介绍）。

自动加载是通过Composer自动处理的，这意味着您可以使用任何PHP类而不做任何额外的事情！ 所有依赖关系都存储在vendor/目录下，但这只是一个惯例。 您可以将它们存储在您想要的任何位置，在您的服务器上或本地在您的项目中。

了解Bundle系统¶

本节介绍Symfony中最强大的功能之一，bundle系统。

bundle有点像其他软件中的插件。 那么为什么它被称为一个bundle，而不是一个插件？ 这是因为Symfony中的所有东西都是bundle，从核心框架功能到为应用程序编写的代码。

您为应用程序编写的所有代码都是以bundle方式组织的。 在Symfony中，一个bundle是一组结构化的文件（PHP文件，样式表，JavaScript，图像等），它们实现了一个功能（一个博客，一个论坛，...），并且可以很容易地与其他人共享开发人员。

Bundles是Symfony的一等公民。 这使您能够灵活地使用封装在第三方包中的预构建特性或分发自己的包。 这使得您可以轻松地选择在您的应用程序中启用哪些功能，并按照您的需要进行优化。 最后，您的应用程序代码与核心框架本身一样重要。

Symfony已经包含一个AppBundle，您可以使用它来开始开发您的应用程序。 然后，如果您需要将应用程序拆分为可重用组件，则可以创建自己的包。

注册一个Bundle¶

应用程序由 AppKernel 类的 registerBundles() 方法中定义的bundle组成。 每个bundle都是一个包含一个描述它的Bundle类的目录： 

// app/AppKernel.php
public function registerBundles()
{
    $bundles = array(
        new Symfony\Bundle\FrameworkBundle\FrameworkBundle(),
        new Symfony\Bundle\SecurityBundle\SecurityBundle(),
        new Symfony\Bundle\TwigBundle\TwigBundle(),
        new Symfony\Bundle\MonologBundle\MonologBundle(),
        new Symfony\Bundle\SwiftmailerBundle\SwiftmailerBundle(),
        new Symfony\Bundle\DoctrineBundle\DoctrineBundle(),
        new Symfony\Bundle\AsseticBundle\AsseticBundle(),
        new Sensio\Bundle\FrameworkExtraBundle\SensioFrameworkExtraBundle(),
        new AppBundle\AppBundle();
    );

    if (in_array($this->getEnvironment(), array('dev', 'test'))) {
        $bundles[] = new Symfony\Bundle\WebProfilerBundle\WebProfilerBundle();
        $bundles[] = new Sensio\Bundle\DistributionBundle\SensioDistributionBundle();
        $bundles[] = new Sensio\Bundle\GeneratorBundle\SensioGeneratorBundle();
    }

    return $bundles;
}

除了已经提到的AppBundle外，还要注意内核还支持Symfony中的其他bundle，比如FrameworkBundle，DoctrineBundle，SwiftmailerBundle和AsseticBundle。

配置一个Bundle¶

每个bundle都可以通过用YAML，XML或PHP编写的配置文件来定制。 看看这个默认的Symfony配置示例： 

# app/config/config.yml
imports:
    - { resource: parameters.yml }
    - { resource: security.yml }
    - { resource: services.yml }

framework:
    #esi:             ~
    #translator:      { fallbacks: ["%locale%"] }
    secret:          "%secret%"
    router:
        resource: "%kernel.root_dir%/config/routing.yml"
        strict_requirements: "%kernel.debug%"
    form:            true
    csrf_protection: true
    validation:      { enable_annotations: true }
    templating:      { engines: ['twig'] }
    default_locale:  "%locale%"
    trusted_proxies: ~
    session:         ~

# Twig Configuration
twig:
    debug:            "%kernel.debug%"
    strict_variables: "%kernel.debug%"

# Swift Mailer Configuration
swiftmailer:
    transport: "%mailer_transport%"
    host:      "%mailer_host%"
    username:  "%mailer_user%"
    password:  "%mailer_password%"
    spool:     { type: memory }

# ...

像framework， twig和swiftmailer这样的每个第一级入口定义了特定bundle的配置。 例如，framework配置FrameworkBundle，swiftmailer配置SwiftmailerBundle。

每个environment都可以通过提供特定的配置文件来覆盖默认配置。 例如，dev环境加载config_dev.yml文件，该文件加载主配置（即config.yml），然后修改它以添加一些调试工具： 

# app/config/config_dev.yml
imports:
    - { resource: config.yml }

framework:
    router:   { resource: "%kernel.root_dir%/config/routing_dev.yml" }
    profiler: { only_exceptions: false }

web_profiler:
    toolbar: true
    intercept_redirects: false

# ...

扩展一个Bundle¶

除了作为一个组织和配置代码的好方法外，一个bundle还可以扩展另一个bundle。 Bundle继承允许您覆盖任何现有的bundle，以便自定义其控制器，模板或其任何文件。

逻辑文件名称¶

当你想从一个包中引用一个文件时，使用下面的表示法：@BUNDLE_NAME/path/to/file; Symfony会将@BUNDLE_NAME解析为包的真实路径。 例如，逻辑路径@AppBundle/Controller/DefaultController.php将被转换为src/AppBundle/Controller/DefaultController.php，因为Symfony知道AppBundle的位置。

逻辑控制器名称¶

对于controllers，您需要使用格式BUNDLE_NAME:CONTROLLER_NAME:ACTION_NAME来引用操作。 例如，AppBundle:Default:index映射到AppBundle\Controller\DefaultController类的indexAction方法。

扩展Bundles¶

如果遵循这些约定，则可以使用bundle inheritance来覆盖文件，控制器或模板。 例如，您可以创建一个包 - NewBundle - 并指定它重写AppBundle。 当Symfony加载AppBundle:Default:index时，它将首先在NewBundle中查找DefaultController类，如果它不存在，则查看AppBundle内部。 这意味着一个bundle可以覆盖另一个bundle的几乎任何部分！

你现在明白为什么Symfony如此灵活？在应用程序间共享bundle，您可以选择在本地也可以选择在整个互联网上。

Using Vendors¶

您的应用程序将依赖于第三方库。这些应该存储在vendor/目录中。 在这个目录中，你不应该理会任何东西，因为它是由Composer专门管理的。 他的目录已经包含了Symfony库，SwiftMailer库，Doctrine ORM，Twig模板系统以及一些其他的第三方库和bundle。

了解缓存和日志¶

Symfony应用程序可以包含多种格式（YAML，XML，PHP等）定义的配置文件。Symfony使用自己的缓存系统，而不是为每个请求解析和合并所有这些文件。 事实上，应用程序配置只是针对第一个请求进行解析，然后编译为存储在app/cache/目录中的普通PHP代码。

在开发环境中，Symfony足够聪明，可以在更改文件时更新缓存。 但在生产环境中，为了加快速度，当您更新代码或更改其配置时，清除缓存是您的责任。 执行此命令清除prod环境中的缓存： 

$ php app/console cache:clear --env=prod

在开发Web应用程序时，事情可能会以多种方式出错。 app/logs/目录中的日志文件会告诉您有关请求的所有信息，并帮助您快速解决问题。

使用命令行界面¶

每个应用程序都附带一个命令行界面工具（app/console），可帮助您维护应用程序。 它提供了通过自动执行繁琐和重复的任务来提高生产力的命令。

运行它不带任何参数，以了解更多有关其功能： 

$ php app/console

--help选项可以帮助您发现命令的用法： 

$ php app/console debug:router --help
