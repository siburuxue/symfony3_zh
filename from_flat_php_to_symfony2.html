Symfony vs 纯PHP¶

为什么Symfony比直接写一个单个文件的PHP程序要好？

如果您从未使用PHP框架，不熟悉MVC哲学，或者只是想知道Symfony的所有宣传是什么，本章就是为您准备的。 你会亲身体会Symfony是可以开发比直接编写原生PHP更快更好的软件而不只是语言上告诉你。

在本章中，您将使用纯PHP编写一个简单的应用程序，然后将其重构为更有条理。 你会经历一段时间，看看为什么网络开发在过去几年发展到现在的地位。

最后，您将看到Symfony如何将您从繁琐的任务中解救出来，并让您重新控制您的代码。

使用纯PHP创建一个简单博客程序¶

在本章中，您将仅使用平面PHP构建令牌博客应用程序。首先，创建一个显示已保存到数据库的博客条目的单个页面。用PHP编写的文件又快又脏： 

<?php
// index.php
$link = mysql_connect('localhost', 'myuser', 'mypassword');
mysql_select_db('blog_db', $link);

$result = mysql_query('SELECT id, title FROM post', $link);
?>

<!DOCTYPE html>
<html>
    <head>
        <title>List of Posts</title>
    </head>
    <body>
        <h1>List of Posts</h1>
        <ul>
            <?php while ($row = mysql_fetch_assoc($result)): ?>
            <li>
                <a href="/show.php?id=<?php echo $row['id'] ?>">
                    <?php echo $row['title'] ?>
                </a>
            </li>
            <?php endwhile ?>
        </ul>
    </body>
</html>

<?php
mysql_close($link);
?>

编写起来很快，执行起来很快，而且随着应用程序的增长，无法维护。 有几个问题需要解决：

  • 无错误检查：如果与数据库的连接失败会怎样？
  • 组织糟糕：如果应用程序增长，这个单一文件将变得越来越难以维护。 你应该在哪里放置代码来处理表单提交？ 你如何验证数据？ 代码去哪里发送电子邮件？
  • 很难重用代码：由于所有内容都在一个文件中，因此无法将该应用程序的任何部分重新用于博客的其他“页面”。

注解

这里没有提到的另一个问题是数据库与MySQL绑定的事实。 虽然这里没有介绍，但Symfony完全集成了专门用于数据库抽象和映射的Doctrine。

展现分离¶

代码可以从将应用程序“逻辑”从编写HTML“呈现”的代码中分离出来： 

<?php
// index.php
$link = mysql_connect('localhost', 'myuser', 'mypassword');
mysql_select_db('blog_db', $link);

$result = mysql_query('SELECT id, title FROM post', $link);

$posts = array();
while ($row = mysql_fetch_assoc($result)) {
    $posts[] = $row;
}

mysql_close($link);

// include the HTML presentation code
require 'templates/list.php';

HTML代码现在存储在一个单独的文件（templates/list.php）中，该文件主要是一个使用类似于模板的PHP语法的HTML文件： 

<!DOCTYPE html>
<html>
    <head>
        <title>List of Posts</title>
    </head>
    <body>
        <h1>List of Posts</h1>
        <ul>
            <?php foreach ($posts as $post): ?>
            <li>
                <a href="/read?id=<?php echo $post['id'] ?>">
                    <?php echo $post['title'] ?>
                </a>
            </li>
            <?php endforeach ?>
        </ul>
    </body>
</html>

按照惯例，包含所有应用程序逻辑的文件 - index.php - 被称为“控制器”。 无论您使用的语言或框架如何，术语控制器都是您会听到的很多词汇。 它只涉及处理用户输入并准备响应的代码区域。

在这种情况下，控制器从数据库准备数据，然后包含一个模板来呈现数据。 如果您需要以某种其他格式呈现博客条目（例如，为JSON格式的 list.json.php），将控制器隔离，您可以轻松更改模板文件。

业务逻辑分离¶

到目前为止，应用程序只包含一个页面。 但是如果第二个页面需要使用相同的数据库连接，甚至是相同的博客文章呢？ 重构代码，以便应用程序的核心行为和数据访问功能被隔离在一个名为model.php的新文件中： 

<?php
// model.php
function open_database_connection()
{
    $link = mysql_connect('localhost', 'myuser', 'mypassword');
    mysql_select_db('blog_db', $link);

    return $link;
}

function close_database_connection($link)
{
    mysql_close($link);
}

function get_all_posts()
{
    $link = open_database_connection();

    $result = mysql_query('SELECT id, title FROM post', $link);
    $posts = array();
    while ($row = mysql_fetch_assoc($result)) {
        $posts[] = $row;
    }
    close_database_connection($link);

    return $posts;
}

小技巧

使用文件名model.php是因为应用程序的逻辑和数据访问传统上被称为“模型”层。 在组织良好的应用程序中，代表您的“业务逻辑”的大多数代码应该存在于模型中（而不是在控制器中）。 与此例不同，模型中只有一部分（或没有）实际上涉及访问数据库。

控制器（index.php）现在非常简单： 

<?php
require_once 'model.php';

$posts = get_all_posts();

require 'templates/list.php';

现在，控制器的唯一任务是从应用程序的模型层（模型）获取数据并调用模板来呈现该数据。 这是模型 - 视图 - 控制器模式的一个非常简单的例子。

布局分离¶

此时，应用程序被重构为三个不同的部分，提供各种优势，并有机会重用不同页面上的几乎所有内容。

代码中唯一无法重用的部分是页面布局。 通过创建一个新的layout.php文件来解决这个问题： 

<!-- templates/layout.php -->
<!DOCTYPE html>
<html>
    <head>
        <title><?php echo $title ?></title>
    </head>
    <body>
        <?php echo $content ?>
    </body>
</html>

模板（templates/list.php）现在可以简化为“扩展”布局： 

<?php $title = 'List of Posts' ?>

<?php ob_start() ?>
    <h1>List of Posts</h1>
    <ul>
        <?php foreach ($posts as $post): ?>
        <li>
            <a href="/read?id=<?php echo $post['id'] ?>">
                <?php echo $post['title'] ?>
            </a>
        </li>
        <?php endforeach ?>
    </ul>
<?php $content = ob_get_clean() ?>

<?php include 'layout.php' ?>

你现在有一个设置，可以让你重用布局。 不幸的是，为了实现这一点，你不得不在模板中使用一些丑陋的PHP函数（ob_start(), ob_get_clean()）。 Symfony使用了一个模板组件，可以很容易地完成这个任务。 你会很快看到它的行动。

添加博客”show”页面¶

博客“列表”页面现在已被重构，以便代码更好地组织和可重用。 为了证明它，添加一个博客“显示”页面，该页面显示由id查询参数标识的单个博客文章。

首先，在model.php文件中创建一个新函数，该文件根据给定的ID检索单个博客结果： 

// model.php
function get_post_by_id($id)
{
    $link = open_database_connection();

    $id = intval($id);
    $query = 'SELECT date, title, body FROM post WHERE id = '.$id;
    $result = mysql_query($query);
    $row = mysql_fetch_assoc($result);

    close_database_connection($link);

    return $row;
}

接下来，创建一个名为show.php的新文件 - 这个新页面的控制器： 

<?php
require_once 'model.php';

$post = get_post_by_id($_GET['id']);

require 'templates/show.php';

最后，创建新的模板文件 - templates/show.php - 以呈现个人博客文章： 

<?php $title = $post['title'] ?>

<?php ob_start() ?>
    <h1><?php echo $post['title'] ?></h1>

    <div class="date"><?php echo $post['date'] ?></div>
    <div class="body">
        <?php echo $post['body'] ?>
    </div>
<?php $content = ob_get_clean() ?>

<?php include 'layout.php' ?>

创建第二页现在非常简单，没有代码被重复。 尽管如此，这个页面还是会介绍一个框架可以为您解决的更多滞后问题。 例如，缺少或无效的code class="docutils literal">id查询参数将导致页面崩溃。 如果这会导致一个404页面被渲染，会更好，但这还不能轻易完成。 更糟糕的是，如果您忘记通过intval()函数清除id参数，那么您的整个数据库将面临SQL注入攻击的风险。

另一个主要问题是每个单独的控制器文件都必须包含model.php文件。 如果每个控制器文件突然需要包含附加文件或执行其他全局任务（例如，强制执行安全措施）会怎么样？ 就目前而言，该代码需要添加到每个控制器文件中。 如果你忘记在一个文件中包含某些东西，希望它不涉及安全性...

“Front Controller”解决方案¶

解决方案是使用前端控制器：处理所有请求的单个PHP文件。 使用前端控制器时，应用程序的URI稍有改变，但开始变得更加灵活： 

Without a front controller
/index.php          => Blog post list page (index.php executed)
/show.php           => Blog post show page (show.php executed)

With index.php as the front controller
/index.php          => Blog post list page (index.php executed)
/index.php/show     => Blog post show page (index.php executed)

小技巧

如果使用Apache重写规则（或等效），则可以删除URI的index.php部分。 在这种情况下，博客展示页面的结果URI将被简单地显示/show。

当使用前端控制器时，单个PHP文件（本例中为index.php）呈现每个请求。 对于博客文章展示页面，/index.php/show将实际执行index.php文件，该文件现在负责根据完整的URI在内部路由请求。 如你所见，前端控制器是一个非常强大的工具。

创建Front Controller¶

您即将在应用程序中迈出一大步。 通过一个文件处理所有请求，您可以集中安全处理，配置加载和路由等。 在这个应用程序中，index.php必须足够聪明才能根据请求的URI呈现博客文章列表页面或博客文章展示页面： 

<?php
// index.php

// load and initialize any global libraries
require_once 'model.php';
require_once 'controllers.php';

// route the request internally
$uri = parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH);
if ('/index.php' == $uri) {
    list_action();
} elseif ('/index.php/show' == $uri && isset($_GET['id'])) {
    show_action($_GET['id']);
} else {
    header('Status: 404 Not Found');
    echo '<html><body><h1>Page Not Found</h1></body></html>';
}

对于组织来说，两个控制器（以前称为index.php和show.php）现在都是PHP函数，并且每个控制器都被移动到单独的文件controllers.php中： 

function list_action()
{
    $posts = get_all_posts();
    require 'templates/list.php';
}

function show_action($id)
{
    $post = get_post_by_id($id);
    require 'templates/show.php';
}

作为前端控制器，index.php承担了一个全新的角色，包括加载核心库和路由应用程序，以便调用两个控制器之一（list_action()和show_action()函数）。 实际上，前端控制器开始看起来很像Symfony处理和路由请求的机制。

小技巧

前端控制器的另一个优点是灵活的URL。 请注意，通过仅在一个位置更改代码，可以将博客帖子显示页面的URL从/showw更改为/read。 之前，整个文件需要重新命名。 在Symfony中，URL更加灵活。

到目前为止，该应用程序已经从单一的PHP文件发展成为一个结构化并允许重用代码的结构。 你应该更快乐，但很不满意。 例如，“路由”系统是反复无常的，并且不会意识到列表页面（/index.php）也可以通过/（如果添加了Apache重写规则）访问。 而且，不是开发博客，而是花费大量时间来处理代码的“架构”（例如路由，调用控制器，模板等）。 需要花费更多的时间来处理表单提交，输入验证，日志记录和安全性。 为什么你必须重新解决所有这些常规问题的解决方案？

初识Symfony¶

Symfony来拯救。 在实际使用Symfony之前，您需要下载它。 这可以通过使用Composer完成，Composer负责下载正确的版本及其所有依赖项，并提供自动加载器。 自动加载器是一种工具，可以在不明确包含包含该类的文件的情况下开始使用PHP类。

在根目录中，使用以下内容创建一个composer.json文件： 

{
    "require": {
        "symfony/symfony": "2.6.*"
    },
    "autoload": {
        "files": ["model.php","controllers.php"]
    }
}

接下来，下载Composer ，然后运行以下命令，该命令将Symfony下载到vendor/目录中： 

$ composer install

除了下载你的依赖，Composer还会生成一个vendor/autoload.php文件，它负责自动加载Symfony框架中的所有文件以及composer.json自动加载部分中提到的文件。

Symfony哲学的核心是应用程序的主要工作是解释每个请求并返回响应。 为此，Symfony提供了Request和Response类。 这些类是正在处理的原始HTTP请求和返回的HTTP响应的面向对象表示。 使用它们来改进博客： 

<?php
// index.php
require_once 'vendor/autoload.php';

use Symfony\Component\HttpFoundation\Request;
use Symfony\Component\HttpFoundation\Response;

$request = Request::createFromGlobals();

$uri = $request->getPathInfo();
if ('/' == $uri) {
    $response = list_action();
} elseif ('/show' == $uri && $request->query->has('id')) {
    $response = show_action($request->query->get('id'));
} else {
    $html = '<html><body><h1>Page Not Found</h1></body></html>';
    $response = new Response($html, Response::HTTP_NOT_FOUND);
}

// echo the headers and send the response
$response->send();

控制器现在负责返回一个Response对象。 为了使这更容易，您可以添加一个新的render_template()函数，顺便说一下，它的行为与Symfony模板引擎非常相似： 

// controllers.php
use Symfony\Component\HttpFoundation\Response;

function list_action()
{
    $posts = get_all_posts();
    $html = render_template('templates/list.php', array('posts' => $posts));

    return new Response($html);
}

function show_action($id)
{
    $post = get_post_by_id($id);
    $html = render_template('templates/show.php', array('post' => $post));

    return new Response($html);
}

// helper function to render templates
function render_template($path, array $args)
{
    extract($args);
    ob_start();
    require $path;
    $html = ob_get_clean();

    return $html;
}

通过引入Symfony的一小部分，该应用程序更加灵活可靠。 该请求提供了一种可靠的方式来访问有关HTTP请求的信息。 具体来说，getPathInfo()方法返回一个干净的URI（总是返回/show而不是/index.php/show）。 因此，即使用户转到/index.php/show，该应用程序也足够智能，可以通过show_action()路由请求。

Response对象在构建HTTP响应时提供了灵活性，允许通过面向对象的接口添加HTTP头和内容。 虽然此应用程序中的回复很简单，但随着应用程序的增长，这种灵活性将带来收益。

Symfony简单示例¶

博客已经走过了很长的路，但它仍然包含了很多这样简单的应用程序的代码。 一路上，您已经使用ob_start()和ob_get_clean()来创建一个简单的路由系统和方法来呈现模板。 如果出于某种原因需要从头开始继续构建这个“框架”，那么至少可以使用Symfony的独立路由和模板组件，它们已经解决了这些问题。

与其重新解决常见问题，你可以让Symfony为你照顾它们。 以下是现在使用Symfony构建的示例应用程序： 

// src/AppBundle/Controller/BlogController.php
namespace AppBundle\Controller;

use Symfony\Bundle\FrameworkBundle\Controller\Controller;

class BlogController extends Controller
{
    public function listAction()
    {
        $posts = $this->get('doctrine')
            ->getManager()
            ->createQuery('SELECT p FROM AcmeBlogBundle:Post p')
            ->execute();

        return $this->render('Blog/list.html.php', array('posts' => $posts));
    }

    public function showAction($id)
    {
        $post = $this->get('doctrine')
            ->getManager()
            ->getRepository('AppBundle:Post')
            ->find($id);

        if (!$post) {
            // cause the 404 page not found to be displayed
            throw $this->createNotFoundException();
        }

        return $this->render('Blog/show.html.php', array('post' => $post));
    }
}

这两个控制器仍然是轻量级的。 每个使用Doctrine ORM库从数据库和Templating组件中检索对象以呈现模板并返回Response对象。 列表模板现在比较简单一些： 

<!-- app/Resources/views/Blog/list.html.php -->
<?php $view->extend('layout.html.php') ?>

<?php $view['slots']->set('title', 'List of Posts') ?>

<h1>List of Posts</h1>
<ul>
    <?php foreach ($posts as $post): ?>
    <li>
        <a href="<?php echo $view['router']->generate(
            'blog_show',
            array('id' => $post->getId())
        ) ?>">
            <?php echo $post->getTitle() ?>
        </a>
    </li>
    <?php endforeach ?>
</ul>

布局几乎完全相同：

<!-- app/Resources/views/layout.html.php -->
<!DOCTYPE html>
<html>
    <head>
        <title><?php echo $view['slots']->output(
            'title',
            'Default title'
        ) ?></title>
    </head>
    <body>
        <?php echo $view['slots']->output('_content') ?>
    </body>
</html>

注解

演示模板留作练习，因为根据列表模板创建应该很简单。

当Symfony的引擎（称为Kernel）启动时，它需要一张映射图，以便根据请求信息知道要执行哪些控制器。 路由配置地图以可读格式提供此信息： 

# app/config/routing.yml
blog_list:
    path:     /blog
    defaults: { _controller: AppBundle:Blog:list }

blog_show:
    path:     /blog/show/{id}
    defaults: { _controller: AppBundle:Blog:show }

现在Symfony正在处理所有普通的任务，前端控制器非常简单。 而且由于它的功能很少，所以一旦创建它就不需要触碰它（如果使用Symfony发行版，甚至不需要创建！）： 

// web/app.php
require_once __DIR__.'/../app/bootstrap.php';
require_once __DIR__.'/../app/AppKernel.php';

use Symfony\Component\HttpFoundation\Request;

$kernel = new AppKernel('prod', false);
$kernel->handle(Request::createFromGlobals())->send();

前端控制器的唯一工作是初始化Symfony的引擎（Kernel）并传递一个Request对象来处理。 Symfony的核心然后使用路由映射来确定要调用哪个控制器。 就像以前一样，控制器方法负责返回最终的Response对象。 其实没有太多其他的东西。

有关Symfony处理每个请求的视觉表示，请参阅请求流程图。

Symfony的优点¶

在即将到来的章节中，您将进一步了解每个Symfony的工作原理以及推荐的项目组织。 现在看看如何将博客从平面PHP迁移到Symfony，从而改善了人生：

  • 您的应用程序现在拥有清晰且一致的代码（尽管Symfony不会强迫您这样做）。 这提高了可重用性，并使新开发人员能够更快地提高项目的生产效率;
  • 您编写的代码的100％适用于您的应用程序。 您不需要开发或维护低级别的实用程序，例如自动加载，路由或渲染控制器;
  • Symfony允许您访问开放源代码工具，如Doctrine和模板，安全，表单，验证和翻译组件（仅举几例）;
  • 由于Routing组件，该应用程序现在享有完全灵活的URL ;
  • Symfony以HTTP为中心的体系结构使您能够访问强大的工具，例如由Symfony的内部HTTP缓存提供动力的HTTP缓存或更强大的工具，如Varnish。 这在后面的章节中都会涉及缓存。

也许最重要的是，通过使用Symfony，您现在可以访问由Symfony社区开发的一整套高质量的开源工具！ 在KnpBundles.com上可以找到很多Symfony社区工具的选择。

更好用的模板引擎¶

如果您选择使用它，Symfony标准配置了名为Twig的模板引擎，可使模板更快地写入并更易于阅读。 这意味着示例应用程序可能包含更少的代码！ 例如，以Twig写的列表模板： 

{# app/Resources/views/blog/list.html.twig #}
{% extends "layout.html.twig" %}

{% block title %}List of Posts{% endblock %}

{% block body %}
    <h1>List of Posts</h1>
    <ul>
        {% for post in posts %}
        <li>
            <a href="{{ path('blog_show', {'id': post.id}) }}">
                {{ post.title }}
            </a>
        </li>
        {% endfor %}
    </ul>
{% endblock %}

相应的layout.html.twig模板也更容易编写：

{# app/Resources/views/layout.html.twig #}
<!DOCTYPE html>
<html>
    <head>
        <title>{% block title %}Default title{% endblock %}</title>
    </head>
    <body>
        {% block body %}{% endblock %}
    </body>
</html>

Twig在Symfony中得到了很好的支持。 虽然PHP模板将始终在Symfony中得到支持，但Twig的许多优点仍将继续讨论。 有关更多信息，请参阅模板章节。
